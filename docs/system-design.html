3 System Design

3.1 System Architecture

3.1.1 Three-Tier Architecture

The Garbage Collection Management System implements a modern three-tier architecture that separates concerns and improves maintainability, scalability, and security:

System Architecture Diagram

┌─────────────────────────────────────────────────────────────┐
│                    PRESENTATION TIER                        │
│  React.js Frontend + TailwindCSS + Axios HTTP Client       │
└─────────────────────────────────────────────────────────────┘
                              │
                              ↓ REST API Calls
┌─────────────────────────────────────────────────────────────┐
│                   APPLICATION TIER                          │
│     Node.js + Express.js + JWT Authentication + Multer     │
└─────────────────────────────────────────────────────────────┘
                              │
                              ↓ Database Queries
┌─────────────────────────────────────────────────────────────┐
│                      DATA TIER                              │
│            SQLite Database + Prisma ORM Layer               │
└─────────────────────────────────────────────────────────────┘

Presentation Tier (Client Layer)

  • Web Browsers: Chrome, Firefox, Safari, Edge with modern JavaScript support
  • User Interfaces: HTML5, CSS3, JavaScript with React.js component architecture
  • Responsive Design: Mobile and desktop compatibility using TailwindCSS
  • Client-side Validation: JavaScript form validation and user input sanitization

Application Tier (Business Logic Layer)

  • Web Server: Node.js with Express.js framework for RESTful API development
  • Application Server: Node.js runtime with event-driven, non-blocking I/O
  • Business Logic: JavaScript modules and functions implementing system workflows
  • Session Management: JWT token handling with secure expiration and refresh
  • Input Validation: Server-side validation and sanitization for security

Data Tier (Database Layer)

  • Database Server: SQLite with Prisma ORM for type-safe database operations
  • Data Storage: Relational database tables with proper normalization
  • Data Access: Prisma client providing auto-generated, type-safe database access
  • Query Optimization: Indexed database design for optimal performance

3.3.2 Non-Functional Requirements

3.3.2.1 Performance Requirements

  • Response Time: Page load time < 3 seconds for normal operations
  • API Performance: Database queries < 1 second for standard queries
  • File Upload: File uploads < 10 seconds for 5MB files
  • Report Generation: Report generation < 30 seconds for standard reports
  • Concurrent Users: Support for 100+ concurrent users
  • Request Throughput: Handle 500 requests/minute
  • Database Transactions: Process 200 transactions/minute
  • User Capacity: Support 10,000+ user accounts
  • Daily Requests: Handle 1,000+ garbage requests per day
  • Storage Capacity: 100GB+ for files and database

3.3.2.2 Security Requirements

  • Authentication: JWT-based authentication with 30-minute timeout
  • Password Security: bcrypt password hashing with salt rounds (10+)
  • Session Management: Secure token storage with automatic refresh
  • Account Lockout: Progressive delay after failed attempts
  • Authorization: Role-based access control (RBAC)
  • API Security: JWT validation on all protected routes
  • Resource Security: Permission checks for specific operations
  • Route Protection: Role-based route guards in frontend
  • Data Protection: Input validation and sanitization
  • SQL Injection Prevention: Prisma ORM with parameterized queries
  • XSS Protection: Content Security Policy and input encoding
  • File Upload Security: Type checking, size limits, and validation
  • HTTPS Enforcement: Secure data transmission in production

3.3.2.3 Reliability Requirements

  • System Availability: 99% uptime during business hours
  • Maintenance Windows: Scheduled during off-peak hours
  • Error Recovery: Automatic error recovery for critical functions
  • Fault Tolerance: Graceful error handling and user feedback
  • Database Recovery: Transaction rollback on failures
  • Retry Mechanisms: Automatic retry for transient errors
  • Data Integrity: Database constraints and validation rules
  • Transaction Consistency: ACID compliance for data operations
  • Backup Procedures: Regular backup and recovery procedures

3.3.2.4 Scalability Requirements

  • Horizontal Scaling: Modular architecture supporting multiple servers
  • Load Balancing: Support for load balancing capabilities
  • Database Scaling: Database read/write splitting support
  • Code Optimization: Optimized code for performance
  • Database Optimization: Database indexing and query optimization
  • Caching Strategies: Caching mechanisms for frequently accessed data

3.3.2.5 Usability Requirements

  • Interface Design: Intuitive navigation and clear information hierarchy
  • Design Consistency: Consistent design language across all pages
  • Responsive Design: Responsive design for mobile and desktop
  • Accessibility: WCAG 2.1 AA compliance for inclusive design
  • Error Messages: Clear error messages and help text
  • Progress Indicators: Progress indicators for long-running operations
  • Undo Capabilities: Undo capabilities for critical operations
  • Contextual Help: Contextual help and tooltips

3.5 Non-Functional Requirements

3.5.1 Quality Attributes

Performance

The system must maintain high performance under varying load conditions with sub-second response times for critical operations and efficient resource utilization.

Security

Comprehensive security measures must protect user data, prevent unauthorized access, and ensure system integrity through multiple layers of protection.

Reliability

The system must maintain high availability with minimal downtime, automatic error recovery, and data consistency guarantees.

Usability

User interfaces must be intuitive, accessible, and efficient, requiring minimal training while supporting diverse user needs and abilities.

Scalability

The architecture must support growth in users, data, and functionality without performance degradation or major redesign.
